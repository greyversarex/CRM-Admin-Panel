## 1. Безопасность

### 1.1 🔴 Уязвимые зависимости (`pnpm audit`)

```
8 vulnerabilities: 5 high | 2 moderate | 1 low
```

| Пакет | Версия | CVE | Путь | Критичность |
|-------|--------|-----|------|-------------|
| `path-to-regexp` | <8.4.0 | GHSA-27v5-c462-wpq7 | `express > router > path-to-regexp` | **High (DoS)** |
| `ip-address` | ≤10.1.0 | GHSA-v2v4-37r5-5v8g | `express-rate-limit > ip-address` | Moderate (XSS в Address6) |
| `@tootallnate/once` | <3.0.1 | GHSA-vpq2-c234-7xj6 | `@google-cloud/storage > teeny-request > http-proxy-agent` | Low |

**Что делать:**
- `path-to-regexp`: обновить Express до версии, которая тянет patched 8.4+. Можно зафиксировать override в `pnpm-workspace.yaml` (`overrides: { "path-to-regexp": ">=8.4.0" }`).
- `ip-address`: обновить `express-rate-limit` до 8.5.0 (уже доступно).
- `@tootallnate/once`: транзитивная, низкий риск, можно игнорировать или удалить `@google-cloud/storage`, если не используется.

### 1.2 🟡 SSRF в webhook-dispatcher

**Файл:** `artifacts/api-server/src/services/webhook-dispatcher.ts:48`

```ts
const resp = await fetch(sub.url, { method: "POST", headers, body, signal: ctrl.signal });
```

`sub.url` приходит из таблицы `webhook_subscriptions` — её заполняет admin через UI. Защиты нет:
- Нет фильтра приватных IP (127.0.0.1, 10.0.0.0/8, 169.254.169.254 — Replit/cloud metadata).
- Нет ограничения на схемы (нет проверки `https://`).
- DNS-rebinding не блокируется.

**Сценарий:**  Скомпрометированный или ошибочно настроенный admin заводит `http://169.254.169.254/latest/meta-data/iam/...` — webhook-dispatcher по cron уйдёт за токенами.

**Сравните с `connectors/acrcloud.ts:51`** — там есть осознанный SSRF allowlist (`Anti-SSRF: разрешаем только официальные хосты ACRCloud`). Для webhook-подписок такой защиты нет.

**Рекомендация:** добавить utility `isPublicHttpsUrl(url)`, проверяющую:
1. Схема === `https:` (или `http:` только для dev).
2. Резолв DNS через `dns.lookup()`, проверка результата на private/loopback/link-local.
3. Запрет `metadata.google.internal`, `169.254.169.254`, `metadata.azure.com`.

Та же проверка нужна в:
- `connectors/spotify.ts` (URL фиксированный — OK)
- `routes/settings.ts` — connector test (admin-supplied URL для тест-вебхука)
- `routes/analytics-ugc-import.ts` — Spotify API (фиксированный — OK)
- `routes/publishing-extras.ts` — PRO registration (фиксированные URL ASCAP/BMI)

### 1.3 🟡 Отсутствие CSRF-защиты

**Файлы:** `app.ts`, `routes/auth.ts`

В `app.ts` нет ни `csurf`, ни кастомного middleware-проверки `Origin/Referer` для state-changing запросов. Защита держится исключительно на:
1. `cookie.sameSite: "lax"` (приличная защита, но не панацея — top-level navigation POST формами обходит).
2. CORS-whitelist (защищает от других доменов).
3. JSON-only API (`Content-Type: application/json` нельзя отправить через `<form>` без preflight).

Для админ-панели за HTTPS этого **обычно достаточно**, но:
- Если есть желание защититься от CSRF на 100%, добавить `Origin`/`Referer` валидацию в middleware.
- Особенно для опасных мутирующих GET (если такие есть — лучше проверить).

```ts
rg -n "router\.(get|delete)" artifacts/api-server/src/routes | grep -iE "delete|reset|approve|reject|disable|enable"
```

### 1.5 ✅ Что сделано хорошо

- **Auth-pipeline** в `routes/index.ts:55–80`: чёткая последовательность — публичные роуты (health/auth/signup/label-invite/storage-upload/ddex-inbound) → `securityPolicy` middleware → `requireAuth` → ролевые `adminOnly` обёртки. Покрытие 100%, ни один защищённый роут не открыт.
- **Параметризованный SQL**: все 50+ найденных `sql\`\`` используют Drizzle template-tag, который безопасно биндит параметры. Один `sql.raw("requester")` (alias-имя в JOIN) — статический, без user-input.
- **Presigned uploads** (`routes/storage-upload.ts`): HMAC-подпись над `objectId|exp|maxBytes`, лимит размера зашит в подпись (нельзя подделать), пред-чек `Content-Length` до открытия write-stream, защита от path-traversal (UUID-регэкс).
- **Multer**: лимиты есть везде (5MB avatar, 25MB KYC, 10MB ingest, 200MB audio через presigned). Disk-storage для крупных импортов (не в RAM).
- **Rate-limit**: глобальный 300 req/min на `/api`, отдельный на `/auth/login`, `/auth/change-password`, `/signup-requests` (3/hour/IP).
- **bcrypt** на паролях, `connect-pg-simple` для сессий в Postgres.
- **Helmet**: включён, CSP в проде с разумной policy. В dev CSP отключён осознанно (Vite HMR + Replit dev banner).
- **CORS-whitelist** через `WEB_ORIGINS` env.
- **Per-account lockout** — упоминается в `replit.md` («per-account lockout», реализован в `securityPolicy`).
- **SESSION_SECRET** — обязательный в проде (бросает ошибку), в dev fallback с warning.

## 4. Архитектура и эксплуатация

### 4.1 ✅ Workflows на Replit

Текущие пять workflow-сервисов запущены и работают:
- `API Server` (порт 8080) — основной API
- `Start application` (порт 5000) — Vite-dev для CRM-панели с прокси `/api → 8080`
- `artifacts/api-server: API Server` — дублирующий, висит с `EADDRINUSE` (см. ниже)
- `artifacts/crm-panel: web` — артефакт-обёртка для canvas
- `artifacts/mockup-sandbox: Component Preview Server` — UI-прототайпер

⚠️ **Дубль:** `artifacts/api-server: API Server` пытается слушать тот же 8080 и падает с `EADDRINUSE`. Это безвредный дубль (один экземпляр уже работает), но захламляет логи. Можно отключить или дать ему другой порт через ENV в `.replit`.

### 4.2 ✅ Graceful shutdown

`src/index.ts:48-68` — корректная остановка всех воркеров (delivery, ack-poller, fraud-engine, payment-automation) до закрытия HTTP-сервера + 10-сек hard-timeout. Образцово.

### 4.3 ✅ Workers

- `delivery-worker` — DDEX-доставка с retry (`MAX_ATTEMPTS`).
- `ack-poller` — опциональный (`DDEX_ACK_POLLER_ENABLED=1`).
- `fraud-engine` — каждые 10 минут.
- `payment-automation` — cron-расписание из БД, перезагрузка раз в минуту.

Все логируют через `pino` со структурированными полями.
