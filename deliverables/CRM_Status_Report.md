## 3. Роли и доступы

### 3.1. Реализованные роли

В таблице `users.role` принимаются 4 значения:

- **`admin`** — полный доступ ко всему
- **`manager`** — операционная роль, без управления настройками системы
- **`label`** — представитель лейбла, видит только своих артистов
- **`artist`** — артист, видит только себя

### 3.2. Двухуровневая защита

**Это полноценная система прав, а не просто скрытие интерфейса.**

**Уровень 1 — Frontend (`lib/permissions.ts`)**

- `canAccess(role, path)` — определяет доступ к роуту
- Меню и кнопки скрываются на основе роли
- ProtectedRoute редиректит на `/` если нет доступа
- Это только UX-слой, **не безопасность**

**Уровень 2 — Backend (`lib/auth.ts`)** — реальная защита

- `requireAuth` — проверка сессии, возвращает 401 если нет
- `requireRole("admin", "manager")` — middleware на уровне роута, 403 при несовпадении
- `getDataScope(req)` — возвращает `{fullAccess, role, artistId, labelId}` для фильтрации SQL
- `resolveScopeFilter(req, queryArtistId, queryLabelId)` — комбинирует то что попросил клиент с тем что разрешает скоуп; если клиент попросил чужое — возвращает null → 403

## 4. Логика релизов

### 4.1. Создание релиза

`POST /api/releases` (доступно admin, manager, label, artist в своём scope) → создаётся со статусом `draft`.

Поля релиза:
- title, releaseType (single/ep/album)
- upc (универсальный код товара)
- artistId, labelId
- genre, releaseDate
- isExplicit, territories[]

### 4.2. Статусы релиза

| Статус | Что значит | Кто переводит |
|---|---|---|
| `draft` | Создан, можно редактировать | автоматически при создании |
| `pending_review` | Отправлен на модерацию | артист / лейбл / менеджер |
| `approved` | Одобрен, можно отправлять в DSP | admin / manager |
| `rejected` | Отклонён | admin / manager (с reason) |
| `delivering` | В очереди доставки | автоматически после кнопки "Deliver" |
| `live` | Опубликован в DSP | автоматически после подтверждения от DSP |

### 4.3. Модерация

1. Артист / лейбл создаёт релиз, добавляет треки / обложку / сплиты
2. Отправляет на review
3. Менеджер / админ открывает `/releases/:id`, проверяет
4. Нажимает Approve или Reject
5. При Reject обязателен `rejectionReason`, релиз возвращается в `draft`
6. Все действия пишутся в `audit_log`

### 4.4. Доставка в DSP

Реализовано полноценно через `delivery worker`:

1. На странице `/releases/:id` (только admin/manager, статус `approved`) — кнопка "Deliver"
2. `POST /api/releases/:id/deliver` создаёт записи в `deliveries` (по одной на каждый таргет: spotify, apple, youtube, deezer и т.д.)
3. Релиз переводится в статус `delivering`
4. Фоновый worker (`workers/delivery.ts`, интервал 30с, batch 10) берёт pending-доставки, генерит XML/CSV через connector, отправляет
5. Записывается результат: `delivered` или `failed` с retry
6. Когда все таргеты `delivered` → релиз → `live`

29 коннекторов в `connectors/`: Spotify, Apple, YouTube, Deezer, Tidal, Amazon, Boom, Yandex Music, ВК и т.д.

## 5. Финансовая система

### 5.1. Учёт доходов

Таблица `transactions` — записи дохода с DSP. Поля: releaseId, trackId, artistId, labelId, dsp (источник), country, period, streams, revenue, currency.

Способы создания:

- **Ручной ввод** — `POST /api/finance/transactions` (admin / manager)
- **Импорт из CSV** — страница `/finance/import` (admin / manager): загружаешь файл от DSP, маппишь поля, система парсит и создаёт транзакции; то что не смогла привязать → в `ingestion_unmatched` для ручного разрешения

### 5.2. Сплиты (распределение)

Таблица `splits` со структурой `{ trackId, recipientUserId, sharePct }`. Сумма долей по треку = 100%.

Расчёт роялти на странице `/royalties`:

1. Берутся все транзакции за период
2. Каждая транзакция мапится на трек → берутся сплиты этого трека
3. Доход транзакции делится по процентам между получателями
4. Возникает баланс на каждом юзере

### 5.3. Выплаты

Таблица `payouts`: `{ userId, amount, currency, status, requestedAt, processedAt, rejectionReason }`.

Поток выплаты:

1. Юзер на `/payouts` нажимает "Запросить выплату" → создаётся запись `pending`
2. **Гард**: должно выполняться `kyc_status='approved'` И заполнены `bank_account_number` + `swift` + `bank_account_holder`. Иначе 403 с понятной причиной
3. Админ/менеджер видит pending → `PATCH /payouts/:id/approve` или `/reject`
4. Approved → `processedAt` ставится, баланс юзера уменьшается

### 5.4. Кто что видит

| Роль | Доступ к финансам |
|---|---|
| Admin / Manager | Все транзакции, все выплаты, могут approve/reject |
| Label | Транзакции своего лейбла и артистов лейбла; свои выплаты |
| Artist | Только транзакции своих треков; свои выплаты |

Запрос с фильтром "чужой artistId" → 403 на бэке (через `resolveScopeFilter`).

## 6. Пользователи и связи

### 6.1. Структура связей

```
labels (1) ──── (N) artists ──── (N) releases ──── (N) tracks
                                                        │
users.role=label ──── label_id (FK)                     │
users.role=artist ──── artist_id (FK)                   │
                                                  splits (recipient_user_id)
```

### 6.2. Реализованные связи

| Связь | Реализована? | Через |
|---|---|---|
| Label → 1 owner_user_id | ✅ | `users.label_id` |
| Label → N artists | ✅ | `artists.label_id` |
| Artist → 1 owner_user_id | ✅ | `users.artist_id` |
| Artist → N releases | ✅ | `releases.artist_id` |
| Release → 1 label | ✅ | `releases.label_id` |
| Release → N tracks | ✅ | `tracks.release_id` |
| Track → split recipients | ✅ | `splits.track_id` → `recipient_user_id` |
| Manager → артисты | ❌ нет explicit-привязки (manager видит всё) |  |

### 6.3. Ответы на типовые вопросы

- **Может ли Label иметь несколько артистов?** Да, через FK `artists.label_id`
- **Привязка релиза к владельцу?** Да, через `release.artist_id` + `release.label_id`. И артист, и лейбл "видят" свой релиз
- **Может ли артист быть без лейбла?** Да, поле `label_id` nullable
- **Может ли релиз иметь нескольких артистов (feat)?** Только основной + featured через теги треков, не как массив на релизе. Для коллабораций функциональность ограничена

Используется **Recharts**: LineChart, BarChart, PieChart, AreaChart.

Источник данных: `usage_reports` (агрегаты для быстрого чтения) + on-the-fly из `transactions`.

- Нет cohort-анализа артистов
- Нет funnel-метрик (релиз → одобрение → live → стримы — нет сводной воронки)
- Нет прогноза (forecast)
- Нет drill-down с графика в детали

- Аудит-лог пишется не везде — нужно пройти и убедиться что все мутации логируются

## 9. Общая оценка готовности

### 9.1. Что работает (production-ready или близко)

| Блок | % готов | Комментарий |
|---|---|---|
| Auth (логин, сессии, роли) | 95% | Нет 2FA и password recovery |

| KYC процесс | 90% | End-to-end готов, требует ручной проверки |

### 9.2. Критично для MVP, но не сделано

1. **Email-нотификации** — пользователь не узнает об одобрении релиза, выплате, KYC, отказе. Нужно SMTP + шаблоны (Mailgun / SES / SendGrid)
2. **Восстановление пароля** — пользователь забыл → потерян
3. **Email-верификация при signup** — спам-заявки
4. **Реальные интеграции с DSP** — connector'ы есть, но нужно подключить настоящие credentials каждого DSP (Spotify for Artists API, Apple Music for Artists и т.д.)

### 9.3. Технически готово к запуску при условии

- Залить seed → 4 тестовых юзера
- Прокликать каждую роль (план тестирования)
- Поправить найденные баги прав
- Подключить хотя бы 1-2 реальных DSP для пилотных релизов
- Накатить SMTP для критичных уведомлений

### 9.4. Итоговая оценка

**~80% готовности к закрытому пилоту** (10-20 артистов, ручное сопровождение менеджером)

**~60% готовности к публичному запуску** (нужен SMTP, password recovery, реальные DSP-credentials, нагрузочное тестирование)
